# Prompt جاهز للإرسال إلى الوكيل البرمجي (Claude Code / أي وكيل آخر)

> انسخ كل ما يلي وأرسله كاملاً للوكيل. لا تحذف أي قسم — كل قسم يمنع مشكلة حقيقية موجودة فعلياً في المستودع.

---

## السياق

أنت تعمل على مستودع `SmartBot` — لوحة تحكم لإدارة صفحة فيسبوك (Backend: FastAPI + SQLAlchemy async، Frontend: React عبر Vite). المستودع منشور حالياً على Render.

**قبل أن تكتب أي سطر كود، افحص هذه الحقيقة الأساسية:** ملفات `App.jsx` و `main.jsx` في جذر المشروع تستورد مكونات وحزماً (`@/components/ui/sidebar`, `@/components/theme-provider`, `@/components/app-sidebar`, `@/pages/dashboard`, `@/pages/rules`, `@/pages/replies`, `@/pages/posts`, `@/pages/settings`, `./index.css`) **غير موجودة إطلاقاً** في المستودع. لا يوجد `package.json`، ولا مجلد `src/`. الناتج الوحيد الموجود هو بناء مُصغّر (minified) في `static/assets` غير قابل للتعديل.

**مهمتك ليست "تعديل" واجهة قائمة — بل بناء واجهة كاملة من الصفر**، باستخدام نفس البنية والمكتبات المذكورة في `App.jsx` كنقطة انطلاق معمارية (React + Vite + shadcn/ui + Tailwind + Framer Motion + TanStack Query + Sonner)، لأن الـ Backend (FastAPI) في `main.py` يعمل فعلياً ويوفر كل الـ API اللازمة (`/api/stats`, `/api/rules`, `/api/replies`, `/api/posts`, `/api/publish`, `/api/bot/status`, `/api/bot/restart`, `/api/logs`) — لا تُعد كتابة الـ Backend، فقط الواجهة والمصادقة.

---

## المتطلبات الحتمية (Blocking — لا تُنشر أي شيء بدونها)

### 1. المصادقة أولاً
لا يوجد حالياً أي نظام تسجيل دخول. أضف:
- صفحة تسجيل دخول (اسم مستخدم/كلمة مرور، مع تجزئة كلمة المرور bcrypt أو ما شابه في الـ Backend).
- جلسة محمية (JWT في httpOnly cookie أو ما يعادلها) — لا localStorage للتوكن.
- Middleware في FastAPI يمنع الوصول لأي نقطة `/api/*` (عدا `/api/login`) بدون جلسة صالحة.
- توجيه تلقائي لصفحة الدخول عند انتهاء الجلسة أو عدم تسجيل الدخول.
- **لا تنشر أي زر حذف/تعديل/نشر جديد في الواجهة قبل أن تكون هذه الطبقة تعمل فعلياً.**

### 2. نظام مستخدمين وأدوار (لأن الطلب هو "كل الصلاحيات")
أضف جدول `User` (id, username, password_hash, role, created_at) بقاعدة البيانات الحالية (SQLAlchemy). أدوار مقترحة كحد أدنى:
- `admin`: كل الصلاحيات (قراءة/إضافة/تعديل/حذف/نشر/تحكم بالبوت).
- `editor`: قراءة + إضافة/تعديل القواعد، بدون حذف وبدون تحكم بالبوت.
- `viewer`: قراءة فقط لكل الصفحات.

كل نقطة API حساسة (DELETE, PUT, POST للقواعد والمنشورات وإعادة تشغيل البوت) يجب أن تتحقق من الدور قبل التنفيذ، وتُخفي الواجهة الأزرار التي لا يملك المستخدم صلاحيتها.

## بناء الواجهة — البنية والصفحات

أنشئ `package.json` كامل ومجلد `src/` بالبنية التالية على الأقل:
```
src/
  index.css
  components/
    ui/            (مكونات shadcn/ui: button, card, table, dialog, sidebar, dropdown, toast...)
    app-sidebar.jsx
    theme-provider.jsx
    kpi-card.jsx
    data-table.jsx   (جدول قابل لإعادة الاستخدام: بحث + فرز + ترقيم صفحات)
    confirm-dialog.jsx
  pages/
    login.jsx
    dashboard.jsx
    rules.jsx
    replies.jsx
    posts.jsx
    settings.jsx
    users.jsx        (إدارة المستخدمين والأدوار — جديدة، لدعم "كل الصلاحيات")
```

### صفحة الدخول (Login)
تصميم بسيط وأنيق، بدون Sidebar، شعار SmartBot، حقل اسم مستخدم/كلمة مرور، رسائل خطأ واضحة بالعربية عند فشل الدخول.

### لوحة القيادة (Dashboard)
- بطاقات مؤشرات (KPI Cards) مع أيقونات وحركة عند التحميل: إجمالي الردود، ردود اليوم، عدد المتابعين، القاعدة الأكثر استخداماً.
- رسم بياني خطي أو مساحي لعدد الردود آخر 7 أيام (استخدم Recharts، متوافق مع RTL).
- رسم بياني إضافي (Bar أو Donut) لتوزيع الردود حسب القاعدة.
- حالة البوت (شغّال/متوقف) كمؤشر حي (Live Badge) مع نبضة (pulse animation) عند التشغيل.
- حالات تحميل (Skeletons وليس نص "جاري التحميل") وحالة فراغ مصممة عند عدم وجود بيانات.

### صفحة القواعد (Rules)
- جدول كامل: بحث نصي فوري، فرز حسب الأعمدة، ترقيم صفحات.
- زر "إضافة قاعدة" يفتح Dialog (وليس صفحة منفصلة) بحقول: الاسم، الكلمات المفتاحية (كـ Tags قابلة للإضافة/الحذف وليس حقل نص فاصل بفواصل)، نص الرد، الوصف.
- تعديل قاعدة موجودة بنفس الـ Dialog مع تعبئة مسبقة للبيانات.
- تفعيل/تعطيل عبر Switch أنيق (ليس زر نصي).
- حذف مع Dialog تأكيد إجباري ("هل أنت متأكد؟ لا يمكن التراجع").
- Toast نجاح/فشل بعد كل عملية (Sonner موجود مسبقاً في التبعيات).

### صفحة سجل الردود (Replies)
جدول مع بحث (باسم المعلّق أو نص التعليق)، فلترة حسب التاريخ، وترقيم صفحات فعلي (Backend يدعمه بالفعل عبر `page` و`per_page`).

### صفحة المنشورات (Posts)
عرض بطاقات للمنشورات الأخيرة + نموذج نشر جديد بمعاينة حية للنص قبل الإرسال + تأكيد قبل النشر الفعلي على صفحة فيسبوك الحقيقية.

### صفحة الإعدادات (Settings)
حالة البوت + تحكم تشغيل/إيقاف/إعادة تشغيل + سجل الأحداث (Logs) بألوان حسب المستوى (خطأ بالأحمر، معلومة بالرمادي) + تبديل الثيم الفاتح/الغامق (يجب أن يعمل فعلياً وليس فقط موجوداً في الكود).

### صفحة المستخدمين (Users) — جديدة
للمدير فقط: قائمة المستخدمين، إضافة مستخدم جديد بدور محدد، تعديل الدور، حذف/تعطيل مستخدم.

## نظام التصميم (Design System) — التزم بهذا ولا تُبدّله لاحقاً

- **اتجاه:** RTL كامل وحقيقي (ليس فقط `dir="rtl"` على `<html>` — تأكد من محاذاة الأيقونات، اتجاه الرسوم البيانية، ومحاذاة الأرقام).
- **الخط:** خط عربي احترافي فعلي (مثل Cairo أو IBM Plex Sans Arabic) محمّل ومطبّق فعلياً على كل النصوص — ليس `system-ui`.
- **الألوان:** عرّف نظام ألوان واحد بصيغة HSL أو Hex قياسية (تجنّب الاعتماد الكلي على `oklch()` لتفادي مشاكل التوافق)، مع نسخة فاتحة وغامقة كاملتين، ومعدلات تباين متوافقة مع WCAG AA للنصوص على الأزرار.
- **الحركة:** استخدم Framer Motion لانتقالات الصفحات (موجود مسبقاً) وحركات دخول خفيفة للبطاقات، بدون مبالغة تُبطئ الاستخدام.
- **لا تُغيّر نظام الألوان أو الخطوط منتصف العمل** — وثّق القرار النهائي في ملف `DESIGN.md` قبل البدء بالتنفيذ، والتزم به حتى النهاية.

---

## تنظيف واجب قبل الانتهاء

1. احذف الملفات المكررة في `static/assets` (النسخ القديمة غير المُشار إليها في `index.html` الحالي).
2. احذف أو انقل إلى `archive/` ملف `templates/index.html` القديم (لوحة HTMX غير المستخدمة) — لا تتركه في المكان الذي قد يُقرأ منه بالخطأ.
3. أضف `.gitignore` يشمل `node_modules/` و`dist/` إن لم تكن موجودة.

## التحقق قبل الإعلان عن الانتهاء (لا تتخطَّ هذه الخطوة)

نفّذ وتحقق من كل ما يلي فعلياً، ولا تكتفِ بالافتراض:
- [ ] `npm install && npm run build` ينجح بدون أي خطأ.
- [ ] لا توجد أخطاء أو تحذيرات في console المتصفح عند فتح كل صفحة (Dashboard, Rules, Replies, Posts, Settings, Users, Login).
- [ ] لا يوجد Overflow أفقي أو عناصر متداخلة على مقاس الجوال (375px) وسطح المكتب (1440px).
- [ ] تسجيل الدخول يعمل فعلياً، ومحاولة الوصول لأي صفحة بدون جلسة تُعيد التوجيه لصفحة الدخول.
- [ ] مستخدم بدور `viewer` لا يرى فعلياً أزرار الحذف/التعديل/النشر، والـ API يرفض الطلب حتى لو استُدعي مباشرة (وليس فقط إخفاء بصري).
- [ ] كل عمليات الحذف تطلب تأكيداً، وكل عملية (نجاح/فشل) تُظهر Toast واضحاً.
- [ ] الثيم الفاتح/الغامق يتبدّل فعلياً بالضغط ويُحفظ عبر إعادة تحميل الصفحة.
- [ ] تشغيل اختبار Playwright بسيط يمر عبر: تسجيل الدخول → فتح كل صفحة → إضافة قاعدة → تعديلها → حذفها، بدون أي فشل.

اعرض لي النتيجة النهائية لكل بند من هذه القائمة، وليس فقط "تم الانتهاء".
